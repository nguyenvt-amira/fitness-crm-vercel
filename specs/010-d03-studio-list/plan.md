## Summary

**FR-001: Studio List Display** is the entry point for all studio management workflows in the D-03 Studio Management module. The feature delivers a fully paginated table of studios scoped by user role and store assignment, with integrated search (by studio name), multi-criteria filtering (store, type, brand, status), and sortable columns. Role-based permissions control data visibility and action affordances (view, edit, delete). Phase 1 delivers a complete, locally-testable UI backed by mock data; Phase 2 integrates the REST API once DEV-BE publishes the stable OpenAPI contract.

## Appendix: Key Decision Rationales

### Why `nuqs` for State Management?

Integrates seamlessly with Next.js App Router; users can bookmark/share filtered views; reduces prop drilling through component tree.

### Why Role-Based Scoping at API Boundary?

Defense in depth; prevents unauthorized data leakage if client-side checks are bypassed; simplifies client logic (no defensive filtering needed).

### Why Centralized Zod Schemas?

Single source of truth for client-side form validation and mock API response validation; avoids duplication.

### Why ≥ 3 Stores for Mock Seed Data?

Allows testing of all role/store assignment combinations; ensures multi-store filtering logic is exercised before Phase 2 backend integration.

### Why Page-Based Pagination Over Infinite Scroll?

Simpler to implement in mock DB; aligns with existing visit-experience-list pattern; easier to test pagination edge cases.
