# SDD Team Protocol — Fitness CRM

> Tool-agnostic development protocol for the Fitness CRM team.
> Every member uses the same spec and task artefacts regardless of which AI tool they run locally.
> Reference: [sdd-dev-workflow.md](./sdd-dev-workflow.md) | [sdd-overview.md](./sdd-overview.md)

---

## 1. Why a Tool-Agnostic Protocol

Team members may use different AI tools (GitHub Copilot, Cursor, Claude, etc.). The protocol ensures:

```
Bad state:
  [no shared docs] → each member prompts their own tool differently
                   → tribal knowledge, tool-specific context, lost decisions

Good state:
  [spec.md + plan.md + tasks.md] → member using GitHub Copilot
                                 → member using Cursor
                                 → member using Claude
  Documents are the source of truth; tools are interchangeable.
```

---

## 2. Information Architecture — Four Quadrants

Requirements and specifications each exist in two modes: **stock** (persistent, system-wide) and **flow** (per-feature).

```
                  STOCK (persistent)                   FLOW (per-feature)
            ┌──────────────────────────────┬──────────────────────────────────┐
            │ System-level business intent  │ What triggered this change        │
 REQUIRE-   │ · Fitness CRM domain rules    │ · "Why this feature now"          │
 MENTS      │ · Gym staff workflows         │ · User stories scoped to feature  │
            │ · Core use cases              │                                   │
            ├──────────────────────────────┼──────────────────────────────────┤
            │ Design map of the system      │ What needs to change              │
 SPECIFI-   │ · Architecture decisions      │ · API contract delta              │
 CATION     │ · Component relationships     │ · UI/UX requirements              │
            │ · Coding conventions          │ · Acceptance criteria             │
            │                              │ · Test scenarios                   │
            └──────────────────────────────┴──────────────────────────────────┘
                   git-tracked · long-lived            git-tracked · per-feature
```

**Stock documents** (`.specify/memory/`, `.github/copilot-instructions.md`) are always loaded by the agent. They change slowly — only when architecture or conventions shift.

**Flow documents** (`specs/<feature>/`) are created per feature and archived after release. They answer "why" (requirements) and "what" (specification) in a single file.

## 3. Protocol Layers

```
Layer 0 — Constitution (.specify/memory/constitution.md)
  Non-negotiable principles I–V. Every agent run must comply.
  Changes require amendment procedure (at least two approvals).

Layer 1 — Copilot Instructions (.github/copilot-instructions.md)
  Active tech stack, project structure, recent feature changes.
  Updated when stack or structure changes.

Layer 2 — Feature Spec (specs/<feature>/spec.md)
  Per-feature "why + what". Generated by speckit.specify, approved by PO/BA.
  Created as draft → moves to approved before implementation begins.

Layer 3 — Implementation Artefacts (plan.md · tasks.md)
  Generated from spec.md by speckit.plan + speckit.tasks.
  Consumed during implementation; temp integration files removed after merge.

Layer 4 — Review Protocol
  Four review gates; each gate has an explicit owner and checklist.
  See Section 6.
```

## 5. Feature Spec Format

Every feature spec lives at `specs/<feature>/spec.md`.

```markdown
# <Feature Name> Spec

> Status: draft | review | approved | implemented
> Branch: ### -<feature-name>
> Related constitution principles: I · II · III · IV · V (mark which apply)

---

## Requirements (Why this change is needed)

### Background

What business event or user need triggered this feature.

### User Stories

- [ ] As a [gym staff role], I want [goal] so that [value].

### Constraints

Deadline · dependencies · out-of-scope items.

---

## Specification (What needs to change)

### Scope

- ✅ In scope:
- ❌ Out of scope (explicitly excluded):

### API Contract

Endpoint · method · request/response shape · error codes.
(Full contract in contracts/api-contracts.md)

### UI Requirements

Screens · states (loading / empty / error / success) · component references.

### Acceptance Criteria

Declarative statements used as the review checklist.

- [ ] Criterion 1
- [ ] Criterion 2

### Connection Points (Do Not Break)

- Existing interfaces that must not change:
- Tables / columns that must not be renamed:
- Shared components being reused:

---

## Constitution Check

| Principle                              | Status                 | Notes |
| -------------------------------------- | ---------------------- | ----- |
| I – Strict Type Safety                 | pass / N/A / exception |       |
| II – Component Purity & UI Consistency | pass / N/A / exception |       |
| III – Server-State via React Query     | pass / N/A / exception |       |
| IV – Schema-Contract Testing           | pass / N/A / exception |       |
| V – Performance Budget                 | pass / N/A / exception |       |

---

## Post-Implementation Notes

Findings to feed back into steering docs or constitution after release.
```

### Spec status lifecycle

```
draft → review → approved → implemented
  ↑        ↑         ↑            ↑
 created  spec PR  implement    feature
          opened   can start     merged
```

**Rule**: Code MUST NOT be committed to the feature branch until `spec.md` status is `approved`.

## 6. Review Gates

### Gate 1 — Spec Approved

**Owner**: PO + BA (with Dev)
**Trigger**: All `[NEEDS CLARIFICATION]` tags resolved; spec PR opened
**Checklist**:

- [ ] All user stories have verifiable acceptance criteria
- [ ] Scope boundaries are explicit (in-scope + out-of-scope)
- [ ] Constitution Check table is filled
- [ ] No `[NEEDS CLARIFICATION]` tags remain

### Gate 2 — Contract Finalized

**Owner**: FE Dev + BE Dev
**Trigger**: BE commits `openapi.json`
**Checklist**:

- [ ] All endpoints in `api-contracts.md` are implemented in `openapi.json`
- [ ] Field names and data types match `data-model.md`
- [ ] Auth flow and error codes match spec
- [ ] Breaking changes have PO sign-off

### Gate 3 — QC Sign-off

**Owner**: QC → PO
**Trigger**: All test cases executed against the integrated branch
**Checklist**:

- [ ] All acceptance criteria in `spec.md` pass
- [ ] All four states tested: loading / empty / error / success
- [ ] No regression in existing features

### Gate 4 — MR Approved

**Owner**: PO / PM (reviewer)
**Trigger**: MR opened against `main` / release branch
**Checklist**:

- [ ] All acceptance criteria are met (verified by QC sign-off)
- [ ] No interfaces in "Connection Points (Do Not Break)" were changed
- [ ] No files outside the feature scope were modified
- [ ] Constitution Check section is `pass` or `N/A` for all five principles
- [ ] `npm run type-check` passes (0 errors)
- [ ] `npm run lint` passes (0 errors)
- [ ] `npm run build` passes (bundle ≤ 250 kB per route)

## 7. Context Injection Per AI Tool

| Tool               | Stock context (always loaded)                                                                                                                              | Flow context (per feature)                                             |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| **GitHub Copilot** | `.github/copilot-instructions.md` auto-loaded. Add `.specify/memory/` + `docs/steering/` files to Copilot context via `#file:` references in agent prompt. | `@` attach `specs/<feature>/spec.md` before invoking any SpecKit skill |
| **Cursor**         | Add `.specify/memory/` + `docs/steering/` + `.github/copilot-instructions.md` to `.cursorrules`                                                            | `@specs/<feature>/spec.md`                                             |
| **Claude**         | Paste `copilot-instructions.md` + constitution + relevant steering files                                                                                   | Paste `spec.md` content                                                |
| **Any chat AI**    | Copy-paste stock context blocks                                                                                                                            | Copy-paste spec.md                                                     |

### Recommended Copilot agent prompt prefix

```
Before starting, read:
- #file:.specify/memory/constitution.md
- #file:docs/steering/_index.md
- #file:docs/steering/architecture.md
- #file:docs/steering/business-glossary.md
- #file:.github/copilot-instructions.md

Do not modify any file listed under "Connection Points (Do Not Break)" in the spec.
Do not proceed past speckit.tasks without running speckit.analyze first.
```

## 8. Update Rules

### Constitution (`.specify/memory/constitution.md`)

| Situation                                         | Action                                                                             |
| ------------------------------------------------- | ---------------------------------------------------------------------------------- |
| New non-negotiable principle needed               | Open proposal issue → draft amendment → two approvals → update version + date      |
| Wording clarification (no semantic change)        | PATCH version bump — single approval sufficient                                    |
| Principle conflicts with real implementation need | Document the exception inline with `TODO(CONSTITUTION-<PRINCIPLE>)` + linked issue |

### Feature Spec (`specs/<feature>/spec.md`)

| Situation                              | Action                                                      |
| -------------------------------------- | ----------------------------------------------------------- |
| Spec error found during implementation | Stop. Open spec PR first. Get approval. Then resume coding. |
| Code change requires spec update       | Update spec in the **same PR** as the code change           |
| Same feature gets a follow-up change   | Create a **new** spec file; link it to the original spec    |

### Copilot Instructions (`.github/copilot-instructions.md`)

| Situation                                  | Action                               |
| ------------------------------------------ | ------------------------------------ |
| New tech added to stack                    | Add to "Active Technologies" section |
| New feature pattern established            | Add to "Recent Changes" section      |
| Structural refactor changes file locations | Update "Project Structure" section   |

### Steering (`docs/steering/`)

Steering files are **additive only** — never remove or rewrite existing content. Each file has its own update trigger:

| File                   | Update when…                                                                                 | Who updates                    |
| ---------------------- | -------------------------------------------------------------------------------------------- | ------------------------------ |
| `_index.md`            | A new module or screen is introduced (add one row to the Module Index table)                 | Agent (speckit.specify) or Dev |
| `architecture.md`      | A new tech pattern, external integration, or structural convention is established            | Dev / Tech Lead                |
| `business-flows.md`    | A new end-to-end flow is added or an existing flow materially changes                        | Agent (speckit.specify) or BA  |
| `business-glossary.md` | A new business term, actor, or domain concept appears that is not yet defined                | Agent (speckit.specify) or BA  |
| `user-personas.md`     | A new user type is introduced or an existing persona's responsibilities significantly expand | BA / PO                        |

**Rule**: `speckit.specify` updates steering automatically as part of step 6.5. Manual updates follow the same additive-only constraint.

## 9. Common Failure Modes and Fixes

| Failure                                                   | Root cause                                      | Fix                                                                                                       |
| --------------------------------------------------------- | ----------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| Agent generates code that contradicts an earlier decision | Spec not loaded before `speckit.implement`      | Always attach `spec.md` before invoking implement                                                         |
| `speckit.analyze` flags CRITICAL constitution violation   | Principle I–V not checked during spec authoring | Fill Constitution Check table in spec before approving                                                    |
| `types.gen.ts` drifts from Zod schemas                    | Codegen pipeline skipped after route change     | Run `generate-openapi` → `generate-api` every time a route or schema changes                              |
| Two `useFiltersHook()` instances cause state desync       | Provider pattern not followed                   | Use a single filter hook in a Provider; child UI reads from context (see `docs/steering/architecture.md`) |
| `speckit.implement` produces code outside feature scope   | Scope not explicitly defined in spec            | Add explicit "Out of scope" section to spec; re-run implement                                             |
| MR blocked by lint errors                                 | `any` or missing types introduced               | Constitution Principle I; run `npm run type-check` + `npm run lint` before opening MR                     |

## 10. Onboarding a New Team Member

1. Read `sdd-overview.md` — understand _why_ SDD is used here
2. Read `.specify/memory/constitution.md` — understand the non-negotiable rules
3. Read `docs/steering/_index.md` and `docs/steering/architecture.md` — system-wide patterns and structure
4. Read `sdd-sequence-flow.md` — understand the full team flow and review gates
5. Read `sdd-dev-workflow.md` — understand the phase-by-phase workflow
6. Browse a completed feature (`specs/staff-management/`) as a reference implementation
7. Set up your AI tool context injection using Section 7 of this document
