# 仕様駆動開発（SDD）— Fitness CRM

> このプロジェクトでSDDを採用する理由と、構造化されていないAI支援コーディングに対してどの課題を解決するのかを整理したドキュメント。

---

## 1. AI支援開発の進化

### フェーズ1 — AIアシスト（2023）

GitHub Copilot や Cursor をインライン補完として利用。人間がコードを書く主体で、AIは高速なオートコンプリートとして機能。

### フェーズ2 — Vibe Coding（2024）

Claude や GPT-4o のようなモデルにより「やりたいことを大まかに説明するだけで動くプロトタイプ」が現実化。非エンジニアでも小規模プロダクトを作れる一方、成長を支える構造は不足。

### フェーズ3 — エージェント型コーディング（2025〜現在）

GitHub Copilot（agent mode）や Cursor が複数ファイルを横断して、自律的に実装・テスト・修正を実行。**この段階でSDDは必須**。構造化された仕様がなければ、エージェントは安定した契約なしで動くことになる。

---

## 2. Vibe Coding が解決したことと破綻ポイント

Vibe Coding は **0→1の速度問題** を解決した。アイデアがあれば、深い技術知識なしでも試作に到達できる。

ただし限界は偶発的ではなく構造的：

| 問題                   | 詳細                                                                                                             |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **コンテキスト喪失**   | 新しいセッションごとに過去判断の記憶がなくなる。コードベースが数百行を超えると、過去方針と矛盾する生成が増える。 |
| **アーキテクチャ不在** | 「とにかく動かす」方針で依存が絡み、後からの機能追加が調査作業化する。                                           |
| **意図の暗黙化**       | 設計理由（なぜ）が記録されない。数か月後に他者や別セッションで根拠を復元できない。                               |
| **正しさの検証不能**   | 「動いて見える」と「仕様通りに正しい」は別。境界条件・セキュリティ・アクセシビリティが体系的に担保されない。     |

### Vibe Coding システムの典型ライフサイクル

```
0–3か月   → 使いやすく、変更も容易
3–12か月  → 変更のたびに副作用、AI依存が拡大
12か月+   → 保守困難、または全面リライトが必要
```

---

## 3. SDD とは何か

> **「AIエージェントへの入力を最適化し、出力を制御可能にする」**

Vibe Coding の問題は入力品質と構造不足に起因する。SDDはその入力を改善する。

### SDD が行う3つのこと

**① 実装前に意図を外部化（仕様先行）**  
受け入れ条件・境界ケース・制約を、実装前に構造化仕様として記述する。

**② コンテキスト境界を定義**  
AIのコンテキストウィンドウに収まる単位へ分解し、各セッションで必要な情報を明示する。

**③ 判断を生きた文書として永続化**  
コードではなく仕様が意図を保持。将来の任意セッションで「なぜこうなっているか」を再構築できる。

### SDD / TDD / BDD の違い

| 手法    | 出発点               | 主な焦点                     |
| ------- | -------------------- | ---------------------------- |
| TDD     | テスト               | ユニットレベルの挙動保証     |
| BDD     | 自然言語シナリオ     | ユーザー視点の振る舞い       |
| **SDD** | **仕様ドキュメント** | **アーキテクチャ制約と契約** |

SDDはTDD/BDDを置き換えない。SDDは上位設計層、TDDはユニット検証層を担当し、相補的に機能する。

## 4. 本プロジェクトにおけるSDD

### ツールチェーン

| ツール                                     | 役割                                                     |
| ------------------------------------------ | -------------------------------------------------------- |
| **GitHub Copilot (agent mode)**            | SpecKitスキルを実行するAIエージェント                    |
| **SpecKit**                                | `specify → clarify → plan → tasks → analyze → implement` |
| **Zod + `@asteasolutions/zod-to-openapi`** | 型とOpenAPIの単一ソース                                  |
| **`@hey-api/openapi-ts`**                  | TypeScriptクライアントとReact Queryファクトリ生成        |
| **TanStack React Query**                   | サーバー状態の管理、生成済みファクトリを通じた取得       |

## 5. SDD成立の3条件

| 条件                           | 状態              | 実現方法                                                                               |
| ------------------------------ | ----------------- | -------------------------------------------------------------------------------------- |
| **仕様が唯一の真実として存在** | ✅                | `specs/<feature>/spec.md` を実装前に生成・レビュー（`draft → approved → implemented`） |
| **コンテキストが制御可能**     | ✅                | フィーチャー単位の自己完結仕様ファイルで必要範囲だけ読ませる                           |
| **仕様とコードが矛盾しない**   | ✅ 新規 / ⚠️ 既存 | 新規は同PRで仕様更新、既存は触れた箇所から段階的に仕様化                               |

---

## 6. SDDを省略した場合のリスク

曖昧な指示のまま `speckit.implement` を実行すると、Vibe Coding の失敗モードを再現する。

| リスク                          | 影響                                                           |
| ------------------------------- | -------------------------------------------------------------- |
| コード生成パイプライン未遵守    | `types.gen.ts` と Zod スキーマが乖離し、実行時型不整合が潜在化 |
| Provider / Context パターン回避 | 状態同期ずれ（URL状態・ページネーション破綻）                  |
| `any` の流入                    | ESLintでビルド停止、Constitution Principle I 違反              |
| 仕様更新なしでコード変更        | 次機能が誤前提で進み、`speckit.analyze` で重大検知             |

---

## 7. まとめ

```
Vibe Coding  → AI活用の初速は高いが制御性を失う
SDD          → 構造化仕様で制御性を取り戻す
              → 境界が明確な新規機能で特に効果的
              → SpecKitが一貫実行のためのスキルチェーンを提供
```

**SDDは万能ではない。境界が明確な機能に正しく適用したときに効果を発揮し、Constitution と Implementation Playbook が逸脱を防ぐ。**
